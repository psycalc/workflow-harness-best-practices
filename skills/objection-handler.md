# SKILL: objection-handler

## Purpose

Preempt and answer the strongest researcher objections without becoming defensive.

## When to Use

- Before outreach
- Before meetings
- When the project risks sounding pseudoscientific, overclaimed, or methodologically weak

## Common Objections

1. `This sounds like pseudoscience.`
2. `How is this different from personality theater?`
3. `Why not just use Big Five / attachment / standard measures?`
4. `Why should simulation predict real outcomes?`
5. `Where is the validation plan?`
6. `Are you building a product story or a research program?`

## Response Strategy

- agree where the objection is fair
- narrow the claim
- show the validation path
- separate heuristics from evidence
- invite falsification rather than certainty

## Output Format

```yaml
objection: "..."
fair_core: "..."
our_response: "..."
boundary: "..."
next_validation_step: "..."
```

## Rules

- Never defend weak claims just because they are already in the repo
- Prefer `yes, that risk is real, here is how we bound it`
- Use epistemic humility as a strength signal
- Show that the project can survive even if part of the typology layer fails
